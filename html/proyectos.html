Desarrollo web

Página de We Colour Estilistas

Se trata del proyecto final de mi curso de Diseño web brindado por Coderhouse. Es una página relativamente sencilla orientada a cubrir el modelo de negocios y dar a conocer distinta información sobre servicios y contacto de la peluquería We Colour Estilistas, situada en la ciudad de Córdoba, Argentina.

Su desarrollo fue lento, ya que obviamente fue donde fuí plasmando todo lo aprendido durante el curso, sin embargo intenté dejarla a un nivel más manejable (hablando del código más que nada), en caso de que se requieran hacer posteriores modificaciones. Ya que después de todo, la idea es que sea una página funcional para complementar la oferta de publicidad de la peluquería.

En esta página di mis primeros pasos en dos herramientas que son las bases del desarrollo web: HTML y CSS. El punto de inflexión en su desarrollo creo yo fue cuando empecé a implementar SCSS (SASS, para ser más específico), ya que mejoró enormemente la la forma de organizar el código CSS hacíendolo mucho más fácil de leer y modificar. Sin duda, una de las cosas que más me costó asimilar fue la correcta separación en media queries para los diseños responsivos, por lo que decidí crear diseños para (además de una versión de escritorio) móvil y una versión para tablets, tratando de mantener la coherencia entre todas las versiones. En la última etapa de desarrollo, decidí incluir un poco de código PHP para el envío de la información que se escriba en el formulario de contacto. Además, decidí dar mi primer paso en JavaScript, creando un archivo para poder animar un carrusel de imágenes el cual se encuentra en la sección de contacto. Tabién es mi primer proyecto gestionado a través de Git y Github.

Desarrollo en Java

XFramework

Se trata de un framework muy sencillo para Java que parsea contenido XML. Si bien la API de Java ya incluye la clase SAXParser para parsear archivos XML, esta sólo nos avisa a través de sus métodos cuándo comienza y cuando finaliza una etiqueta, siendo nosotros quienes programemos el comportamiento que tendrá la aplicación dependiendo del contenido de las etiquetas, lo que se vuelve una tarea pesada si deseamos parsear archivos grandes y repetitiva si parseamos contenido XML en nuestras aplicaciones de manera habitual.

XFramework

Repositorio GitHub

XFramework viene a dar solución a dicho problema con un planteamiento sencillo: esto es, usando la propia API de Java parasea el archivo que le indiquemos a través de la clase XMLFactory, para luego proceder a almacenar toda la información a través de un diseño orientado a objetos que simula la estructura del archivo XML guardando la información de cada tag en un objeto XTag, manteniendo todas las relaciones tag padre/hijo a través de sus atributos. Dichas relaciones pueden apoyarse en un objeto HashtableLinkeado, el cual consiste en una Hastablede listas enlazadas para evitar ambigüedades a la hora de buscar tags con múltiples ocurrencias. Por último, decidí incluir un XML con información genérica para que pueda servir como ejemplo ó para testear y practicar con el framework, dicho XML es sample.xml, el cual incluye tags con múltiples ocurrencias y que contienen atributos.

Nota: este framework NO es de mi autoría, lo saqué del capítulo 12 del libro "Java a Fondo" de Pablo Sznajdleder del cual hablo en la sección de links, yo sólo lo transcribí en un proyecto, lo documenté y lo comprimí en un .jar para posteriormente poder subirlo y usarlo en proyectos propios.

Aplicación cliente - servidor - base de datos

Se trata del proyecto final que desarrollé para mi curso de Java Inicial. Son en realidad dos aplicaciones, un cliente a través del cual un usuario (luego de loguearse) obtiene acceso a funciones básicas de un sistema de gestión y envío de trámites por medio de una interfaz sencilla en la cual aparecen las opciones para crear y finalizar trámites según los permisos que tenga éste usuario. Si bien, por tratarse de un curso de Java inicial el trabajo requerido era en sí mucho más sencillo (la lógica de negocios era la misma, la diferencia es que la aplicación a desarrollar sólo tenía que simular el proceso), decidí ir un paso más adelante y desarrollar una aplicación que realmente cubra esa necesidad.

La aplicación cliente si bien corre completamente en la máquina del usuario, resuelve todas sus peticiones a través de conexiones a un servidor conectado a la misma red, siendo la ventana de login la primera de todas a partir de la cual se genera un objeto UserDTO el cual su utiliza como parámetro para la construcción de la interfaz de usuario dearrollada íntegramente a partir de la librería swing. Una vez recolectados los datos de usuario, dicha interfaz creará las pestañas que separan todas las funciones según los permisos que se tengan y por cada operación en la cual se requiera la interacción con la base de datos se abrirá una conexión al servidor gestionado por la clase utilitaria ServicesLocator, la cual según el método requerido le enviará un código al servidor para poder establecer el flujo de envío de datos. Las "conexiones" entre clientes en realidad se hacen a través de su interacción con el servidor, siendo el envío de información a través de actualizaciones en, por ejemplo, la bandeja de entrada que muestra todos los trámites que fueron enviados al área que le corresponde al usuario. Para mantener actualizada esta información, la gui tiene una clase interna que se encarga de, cada cierto período de tiempo medido en milisegundos, refrescar los componentes sensibles a cambios a través de un hilo independiente que se encarga de hacer peticiones al servidor.

Por su lado el servidor se encarga de (una vez iniciado) escuchar peticiones a través de un ServerSocket que al recibir una conexión crea un hilo el cual (según el "código" correspondiente al servicio que se esté pidiendo) se encargará de establecer un flujo cliente-servido mientras el socket vuelve a seguir a la espera de nuevas conexiones. Una vez establecido el o los servicios requeridos por el cliente, se recurre a un pool de conexiones el cual gestiona las conexiones a la base de datos HSQL, contando con métodos sincronizados para evitar posibles problemas al ejecutar commits cuando se quieren realizar updates (las consultas simples no suelen generar este tipo de problemas).

Base de datos HSQLDB

Sección en GitHub

Si bien la idea del proyecto era representar la lógica de negocios con un diseño orientado a objetos, en esta versión se ve aplicada directamente en la base de datos por medio de las relaciones entre tablas. La base de datos en cuestión es de código abierto y es una HSQLDB, desarrollada íntegramente en Java. El criterio de elección para esta base es que es la única que sé usar a este nivel. Todas las operaciones relacionadas a las peticiones y modificaciones se hacen a través de objetos de transferencia de datos (DTO), los cuales son usados por los objetos de acceso a datos (DAO). Estos DAOs implementan interfaces que describen su comportamiento ya que desde un principio fue mi intención dejar la puerta abierta a la implementación de nuevas bases de datos las cuales podrían requerir de una sintáxis distinta, por lo que me pareció conveniente el describir su comportamiento de antemano.

Un punto interesante de este proyecto es el uso de XFramework para acceder a la parametrización de información relativa tanto a la conexión entre las partes (IP y puerto), como a otros parámetros internos del pool de conexión en el archivo de configuración del servidor y la taza de actualización de información en la configuración del cliente sin la necesidad de recompilar nada. Otra ventaja de este modelo que me gustaría implementar en un futuro es la parametrización de todo el texto, haciendo de esta manera la internacionalización de la aplicación mucho más fácil.

Si bien el proyecto es, dentro de todo funcional, aún está incompleto. Por lo que tengo planeado continuarlo hasta que tenga un aspecto más "profesional", además de ciertos cambios respecto a la versión publicada:

- Actualmente me encuentro estudiando el uso de Hibernate para Java, por lo que seguramente termine usándolo por lo que descartaría ó modificaría buena parte de la clases relacionadas al acceso a la base de datos.
- La forma en la que el servidor gestiona las conexiones con los clientes se siente un poco rudimentaria, por lo que me inclino más a encontrar alguna forma más eficiente y moderna para esto.
- La base de datos HSQLDB se debe iniciar desde consola (iniciar la base de datos antes que el servidor es mandatorio para el correcto funcionamiento de la aplicación) y creo firmemente que es posible solventar esto haciendo uso de la API de introspección de clases provista por Java, área en la que todavía no cuento con muchos conocimientos, pero veo gran utilidad.
- Dentro de la aplicación servidor se encuentra la clase FacadeLocal, la cual está incompleta y fue pensada para proveer servicios a una gui que sería la encargada de proveer la función de "administrador", el cual se encargaría de gestionar claves y usuarios entre otras cosas. La razón de por qué esto nunca vio la luz es por la falta de tiempo que me llevó a prescindir de estas funciones, teniéndo que manejarlas desde la base de datos directamente.

Dicho todo esto, la aplicación fue testeada siendo el servidor hosteado desde mi notebook y corriendo un cliente se pudo realizar operaciones de envíom creación y finalización de trámites a otra aplicación cliente en otra computadora, comunicadas ambas a través de una red inalámbrica de un celular.
